üåê Understanding Web Applications

Key Principle: The larger and more accessible a program is, the higher the risk of security vulnerabilities. Web applications are inherently exposed to millions of potential attackers.

Static vs. Dynamic Webpages

Static Webpage Example

<!DOCTYPE html>
<html>
<head>
    <title>Welcome Page</title>
</head>
<body>
    <h1>Welcome to Our Website</h1>
    <p>This content never changes.</p>
</body>
</html>

Characteristics: Same content for all users, no server-side processing, created with HTML only.

Dynamic Webpage Components

Dynamic webpages adapt content based on user input, time, database queries, or user identity. They require:

  • HTML Forms - Collect user input
  • Server-Side Scripts - Process data (PHP, ASP.NET, JSP)
  • Database Connections - Store and retrieve information
  • Client-Side Scripts - Enhance user experience (JavaScript, AJAX)

Web Forms

Web forms use the <form> element to collect user input and submit it to a web server for processing.

Common Gateway Interface (CGI)

What is CGI?

CGI is an interface standard that determines how a web server passes data to a web browser. It enables dynamic webpage creation using scripting languages like Perl, Python, or C.

#!/usr/bin/perl
# Simple CGI script in Perl
print "Content-type: text/html\n\n";
print "<html><body>";
print "<h1>Hello Security Testers!</h1>";
print "</body></html>";

Main Role: Passing data between web servers and web browsers, enabling dynamic content generation.

üîì Web Application Vulnerabilities

‚ö†Ô∏è Critical Understanding: Application security is as important as network security. Attackers controlling a web server can deface websites, steal data, compromise accounts, and pivot to internal networks.

Common Vulnerability Categories

Input Validation Vulnerabilities

Input validation involves filtering, rejecting, or sanitizing user input before processing. Poor input validation leads to:

SQL Injection CRITICAL

Injecting SQL commands through input fields

Command Injection CRITICAL

Executing OS commands through the application

XSS (Cross-Site Scripting) HIGH

Injecting malicious scripts into web pages

Path Traversal HIGH

Accessing files outside intended directories

SQL Injection Deep Dive

Understanding SQL Injection

SQL Injection (SQLi) occurs when an attacker supplies SQL commands through application input fields.

// Vulnerable SQL Query
SELECT * FROM customer 
WHERE tblusername = '' OR 1=1 -- ' 
AND tblpassword = '' ''

// The injected input
Username: ' OR 1=1 --
Password: (anything)

// Result: Authentication bypassed!
// Because 1=1 is always TRUE
// -- comments out the password check
Testing for SQL Injection:
  • Enter punctuation marks in input fields
  • Try single quotes followed by SQL keywords (SELECT, DROP, etc.)
  • Look for database error messages
  • Test for "Blind SQL Injection" when no errors are shown

Authentication & Authorization Flaws

Authentication Testing

Key questions to ask during authentication testing:

  • Is data passed in cleartext or encrypted?
  • Are passwords stored securely (hashed and salted)?
  • Is the authentication server properly configured?
  • Can authentication be bypassed?
  • Are session tokens properly generated and protected?

Authorization Testing

Authorization checks whether a user has privileges to access resources. Common issues:

  • Reliance on hidden fields or obscured URLs for access control
  • Lack of server-side privilege verification
  • Horizontal privilege escalation (accessing other users' data)
  • Vertical privilege escalation (gaining admin rights)

Error Handling & Information Disclosure

Insecure Error Handling

Dangerous Example: 
// Error message revealing sensitive information
Fatal error: Uncaught mysqli_sql_exception: 
Access denied for user 'webapp_user'@'localhost' 
(using password: YES) in /var/www/html/login.php:15

Stack trace:
#0 /var/www/html/login.php(15): mysqli->__construct()
#1 {main}
Secure Alternative: 
// Generic error message
An error occurred while processing your request.
Please try again later or contact support if the problem persists.

Best Practices:

  • Disable debugging mode in production
  • Log detailed errors server-side only
  • Display generic messages to users
  • Never expose stack traces, database info, or file paths

Cryptography Issues

Common cryptography problems in web applications:

  • Weak Random Number Generators: Predictable session tokens
  • Outdated Algorithms: MD5, SHA-1, DES still in use
  • Self-Signed Certificates: No trusted certificate authority validation
  • Unencrypted Channels: Sensitive data transmitted over HTTP
  • Poor Key Management: Hardcoded keys in source code

Client-Side Vulnerabilities

Critical Principle: Never trust client-side security controls alone!

Common Client-Side Issues:

  • Sensitive data stored in cookies or local storage
  • Client-side input validation only (can be bypassed)
  • JavaScript-based access controls
  • Exposed API keys and credentials in client code

üîç Web Application Testing Methodologies

Testing Approaches

SAST - Static Application Security Testing

White Box Testing

Analyzes source code for vulnerabilities without executing the program

Pros: Finds vulnerabilities early, pinpoints exact location

Cons: Requires source code access, can have false positives

DAST - Dynamic Application Security Testing

Black Box Testing

Tests running applications from the outside, simulating real attacks

Pros: No source code needed, finds runtime issues

Cons: Later in development cycle, limited code coverage

IAST - Interactive Application Security Testing

Gray Box Testing

Combines SAST and DAST using agents inside the application

Pros: Real-time analysis, fewer false positives

Cons: Requires agent installation, performance overhead

Platform Security & Configuration Testing

Configuration Assessment Checklist

  1. Identify the web server software and version
  2. Check for known vulnerabilities in server and components
  3. Verify encryption is used for sensitive communications
  4. Test for administrative interfaces and default credentials
  5. Review server configuration files for misconfigurations
  6. Check for unnecessary services and open ports
  7. Verify proper file permissions and directory listings
  8. Test for information disclosure in HTTP headers

Business Logic Testing

Understanding Business Logic Flaws

Business logic refers to procedures users follow to accomplish goals. Logic flaws allow bypassing these procedures.

Example Scenarios:

  • Price Manipulation: Changing prices in POST data before checkout
  • Quantity Bypass: Ordering negative quantities for refunds
  • Workflow Bypass: Skipping required approval steps
  • Rate Limiting: Circumventing transaction limits
  • Race Conditions: Exploiting timing dependencies

Burp Suite

Burp Suite Professional Features

Industry-standard web application security testing platform included in Kali Linux.

Key Features:
  • Intercepting Proxy: Capture and modify HTTP/HTTPS traffic between browser and server
  • Spider: Automatically crawl web applications to map structure
  • Scanner: Automated vulnerability detection (Professional version)
  • Intruder: Automated attack tool for fuzzing and brute force
  • Repeater: Manually modify and resend individual requests
  • Decoder: Encode/decode data in various formats
  • Comparer: Visual diff tool for comparing requests/responses

Wapiti

Black Box Web Application Vulnerability Scanner

Wapiti performs "black box" testing by crawling pages and injecting payloads to detect vulnerabilities.

Detection Capabilities:
  • SQL Injection (SQLi)
  • Cross-Site Scripting (XSS)
  • File Inclusion (LFI/RFI)
  • Command Execution
  • XXE (XML External Entity)
  • CRLF Injection
  • Server-Side Request Forgery (SSRF)
Wapiti Usage Examples
# Basic scan wapiti -u http://target.com # Scan with authentication wapiti -u http://target.com --auth-cred "user:pass" # Test specific modules wapiti -u http://target.com -m sql,xss # Generate HTML report wapiti -u http://target.com -f html -o /tmp/report # Verbose output with all modules wapiti -u http://target.com -v 2 -m all
Note: Wapiti uses aggressive testing techniques. Only use on applications you have explicit authorization to test.

Lab 1: Web Application Reconnaissance

Objective: Gather information about a web application's architecture and technology stack.

Setup:

Lab 2: SQL Injection Testing

Objective: Identify and exploit SQL injection vulnerabilities using manual techniques.

Prerequisites:

  • DVWA or similar vulnerable application
  • Basic SQL knowledge
  • Burp Suite or OWASP ZAP configured

Procedure:

  1. Navigate to the SQL Injection page in DVWA (set security to "low")
  2. Test input field with single quote (') to check for errors
  3. Try basic SQLi payloads: ' OR 1=1 --

  5. Extract database names using enumeration techniques

Lab 3: Cross-Site Scripting (XSS) Detection

Objective: Identify and demonstrate reflected, stored, and DOM-based XSS vulnerabilities.

Procedure:

  1. Navigate to XSS testing page in vulnerable application
  2. Test for reflected XSS with: <script>alert('XSS')</script>
  3. Try alternative payloads if basic ones are filtered
  4. Test for stored XSS in comment/message fields
  5. Use Burp Suite Intruder with XSS payload list
  6. Document successful payloads and their contexts
